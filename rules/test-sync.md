# 🧪 Test Sync Rules

코드를 변경할 때는 영향받는 테스트도 반드시 함께 수정해야 합니다. 테스트가 코드 변경을 따라가지 못하면 거짓 실패(false negative)가 발생합니다.

## 1. 코드 변경 시 테스트 동기화 의무

코드를 수정하기 전, 아래 순서를 반드시 따릅니다.

1. **영향 범위 파악:** 변경할 파일을 수정하기 전에 해당 파일을 import하거나 의존하는 테스트 파일을 먼저 검색합니다.
   ```bash
   grep -r "변경할파일명" src --include="*.test.*" --include="*.spec.*"
   ```
2. **테스트 선수정:** 변경된 인터페이스(함수 시그니처, 응답 구조, 타입 등)에 맞게 테스트를 먼저 수정합니다.
3. **본 코드 수정:** 테스트 수정 후 실제 코드를 변경합니다.
4. **전체 테스트 실행:** `npx vitest run`으로 모든 테스트가 통과하는지 확인합니다.

## 2. 변경 유형별 동기화 범위

| 변경 유형 | 동기화 대상 |
|-----------|------------|
| 함수 시그니처 변경 (파라미터 추가/제거/순서 변경) | 해당 함수를 mock하거나 직접 호출하는 모든 테스트 |
| API 응답 구조 변경 (`data` 래퍼 추가 등) | MSW 핸들러 테스트, 해당 API를 호출하는 통합 테스트 |
| 타입 변경 (필드 추가/제거/이름 변경) | 해당 타입을 사용하는 테스트의 mock 데이터 |
| 파일 경로/이름 변경 | import 경로가 깨진 테스트 |
| 상수값 변경 | 해당 상수를 하드코딩으로 검증하는 테스트 |

## 3. 절대 허용하지 않는 행위

- 테스트가 실패한다고 해서 **테스트를 삭제하거나 skip** 처리하는 것
- 코드만 변경하고 **테스트는 나중에 수정**하겠다고 미루는 것
- 테스트를 통과시키기 위해 **본 코드의 의도를 바꾸는** 것
